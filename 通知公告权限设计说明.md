# 通知公告权限设计说明

## 设计目标

实现学校端通知公告管理功能，满足以下需求：
1. 学校可以查看管理员发布的公告（只读）
2. 学校可以发布、编辑、删除本校的公告
3. 学校发布的公告可以指定目标受众（教职工/学生）

---

## 当前实现方案（简化版）

### 核心思路
利用现有的 `sys_notice` 表，通过 `create_by` 字段区分公告来源：
- `create_by = 'admin'` → 管理员发布的公告
- `create_by = school_id` → 学校发布的公告

### 优点
1. **无需修改数据库表结构**，开发速度快
2. **实现简单**，适合毕业设计项目
3. **功能完整**，满足基本需求

### 缺点
1. 无法在数据库层面区分目标受众（教职工 vs 学生）
2. 查询逻辑稍复杂，需要在 SQL 中判断 `create_by`
3. 扩展性较差，后续如需添加更多权限类型需要重构

### 实现细节

#### 1. 列表查询
```sql
SELECT * FROM sys_notice
WHERE 1=1
  -- 显示管理员发布的已发布公告
  AND (create_by = 'admin' AND status = 1)
  -- 或显示本校发布的所有公告（包括草稿）
  OR (create_by = #{schoolId})
ORDER BY is_top DESC, publish_time DESC;
```

#### 2. 权限判断（前端）
```typescript
// 判断是否为管理员公告
const isAdminNotice = (row) => {
  return row.createBy === 'admin' || row.createBy !== currentSchoolId
}

// 根据判断结果控制操作按钮
<template v-if="!isAdminNotice(row)">
  <el-button @click="handleEdit(row)">编辑</el-button>
  <el-button @click="handleDelete(row)">删除</el-button>
</template>
```

#### 3. 权限校验（后端）
```sql
-- 更新公告时校验权限
UPDATE sys_notice
SET notice_title = #{noticeTitle}, ...
WHERE notice_id = #{noticeId}
  AND create_by = #{schoolId};  -- 只能更新本校公告

-- 如果影响行数为 0，说明无权限或公告不存在
```

#### 4. 目标受众处理
- 前端表单提供 `targetAudience` 选项（教职工/学生）
- 后端暂不处理此字段，或存储在备注字段中
- 待数据库扩展后再实现真正的受众过滤

## 推荐扩展方案（生产级）

### 数据库扩展

如果需要更完善的权限管理，建议添加以下字段：

```sql
-- 扩展 sys_notice 表
ALTER TABLE sys_notice ADD COLUMN publisher_type VARCHAR(20);  
-- 发布者类型：admin=管理员, school=学校

ALTER TABLE sys_notice ADD COLUMN publisher_id VARCHAR(255);   
-- 发布者ID：对于学校发布的公告，存储 school_id

ALTER TABLE sys_notice ADD COLUMN target_audience VARCHAR(20); 
-- 目标受众：all=所有人, school=学校教职工, student=学生

-- 添加索引
CREATE INDEX idx_notice_publisher ON sys_notice(publisher_type, publisher_id);
CREATE INDEX idx_notice_audience ON sys_notice(target_audience);

-- 更新现有数据
UPDATE sys_notice SET publisher_type = 'admin', target_audience = 'all' WHERE create_by = 'admin';
```

### 权限逻辑

#### 1. 管理员发布
```json
{
  "publisher_type": "admin",
  "publisher_id": null,
  "target_audience": "all"
}
```
- 所有学校和学生都可见
- 用于发布省级政策、通知等

#### 2. 学校发布（面向教职工）
```json
{
  "publisher_type": "school",
  "publisher_id": "school_001",
  "target_audience": "school"
}
```
- 仅本校教职工可见
- 用于内部管理通知

#### 3. 学校发布（面向学生）
```json
{
  "publisher_type": "school",
  "publisher_id": "school_001",
  "target_audience": "student"
}
```
- 本校教职工和学生都可见
- 用于学生相关通知

### 查询逻辑

#### 学校端查询
```sql
SELECT * FROM sys_notice
WHERE status = 1  -- 只显示已发布的
  AND (
    -- 管理员发布的全局公告
    (publisher_type = 'admin' AND target_audience = 'all')
    -- 或本校发布的公告
    OR (publisher_type = 'school' AND publisher_id = #{schoolId})
  )
ORDER BY is_top DESC, publish_time DESC;
```

#### 学生端查询（移动端）
```sql
SELECT * FROM sys_notice
WHERE status = 1
  AND (
    -- 管理员发布的全局公告
    (publisher_type = 'admin' AND target_audience = 'all')
    -- 或本校发布的面向学生的公告
    OR (publisher_type = 'school' AND publisher_id = #{schoolId} AND target_audience = 'student')
  )
ORDER BY is_top DESC, publish_time DESC;
```

### 优点
1. **权限清晰**，数据库层面即可控制可见性
2. **扩展性强**，后续可以添加更多受众类型（如：特定学院、特定专业）
3. **查询高效**，通过索引优化查询性能
4. **易于维护**，权限逻辑集中在数据库层

### 缺点
1. 需要修改数据库表结构
2. 需要迁移现有数据
3. 开发工作量稍大

## 实施建议

### 阶段一：毕业设计阶段（当前方案）
- 使用简化版方案，通过 `create_by` 字段区分来源
- 前端控制操作权限，后端 SQL 校验权限
- `targetAudience` 字段仅前端使用，不影响实际权限
- **优势**：快速完成功能，满足答辩需求

### 阶段二：答辩后优化（可选）
- 如果项目需要继续开发或投入使用，再进行数据库扩展
- 实施推荐扩展方案，完善权限管理
- 迁移现有数据，更新前后端代码
- **优势**：生产级方案，易于维护和扩展

## 前端实现要点

### 1. 权限判断
```typescript
// 当前方案
const isAdminNotice = (row) => {
  return row.createBy === 'admin' || row.createBy !== userStore.userInfo?.ownerId
}

// 扩展方案
const isAdminNotice = (row) => {
  return row.publisherType === 'admin'
}

const canEdit = (row) => {
  return row.publisherType === 'school' && row.publisherId === currentSchoolId
}
```
